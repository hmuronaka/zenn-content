# はじめに

SwiftUIを用いて、1年以上[cut & loop](https://apps.apple.com/jp/app/cut-loop-%E3%83%AA%E3%82%B9%E3%83%8B%E3%83%B3%E3%82%B0-%E3%83%97%E3%83%AC%E3%82%A4%E3%83%A4%E3%83%BC/id616310281)という個人アプリを開発してきました。

現在は、iOS開発に関してはcut & loopの改善と個人アプリの開発くらいでiOS開発からは離れているため、こうした情報を発信する機会がないため、SwiftUIで遭遇したバグの記録とその対処方法について、共有したいと思います。

# iOSバージョン, 互換性の問題など

SwiftUIを継続利用して、一番遭遇した問題はiOSのバージョンが変わるごとにAPIが変わることによる問題です。

例えばiOS15の[NavigationView](https://developer.apple.com/documentation/swiftui/navigationview)は、iOS16で[NavigationStack](https://developer.apple.com/documentation/swiftui/navigationstack/)/[NavigationSplitView](https://developer.apple.com/documentation/swiftui/navigationsplitview)に変わりました。

またiOS16までの[ObservableObject](https://developer.apple.com/documentation/combine/observableobject)は、iOS17から[Observation](https://developer.apple.com/documentation/observation)が導入されます。

マイナーバージョンアップでもAPIの追加、内容の変更が行われ、そこにバグが含まれることも多いです。

以下では主にiOSバージョンが変わったタイミングで遭遇したバグや、互換性の問題を紹介します。

----

## バグ１ iOS16.4以降 NavigationStackの遷移先ビューと@Environment(\.dismiss)と合わせて利用した場合、NavigationLinkをタップすると、NavigationStackの内部処理が無限ループに陥り処理続行不能になる

[Apple Developer Forum iOS 16.4 NavigationStack Behavior Unstable](https://developers.apple.com/forums/thread/727282)

### 事象

iOSが16.4にアップデートされた後に、cut & loopで遭遇したバグです。（iOS16.5でも再現します。）

Navigationの下位ビューに遷移しようとセルをタップすると画面が固まり、復旧はアプリの再起動するしかない、というバグです。
発生条件は不明で(@Environment(\.dismiss)が関係している？）、発生件数も少なさそうですが、遭遇するととても厄介。

再現コードは上記リンクに掲載していますが、その再現コードではAdMobがリンクされていないと再現しません。（別の方のコメントではAdMobがなくとも発生しているようです。）
View構造によって発生したり、しなかったり、また回避策の点からもSwiftUIがメモリ破壊ならぬフレームワーク破壊的な振る舞いを起こしていると感じています。

### 原因

[iOS16.4のリリースノート](https://developer.apple.com/documentation/ios-ipados-release-notes/ios-ipados-16_4-release-notes)を確認するとNavigationStack/NavigationSplitView周りでの変更が行われてたため、それが影響しているかもしれません。

またこの事象は、iOS15のNavigationViewでの勝手画面がpopされるバグ(後述）を思い出させるため、私はこれはデグレの一種ではないかと想像しています。

### 対応方法

再現をさせるために新規プロジェクトを作成し再現を試みたのですが、全く再現せずに手を焼きました。結局、cut & loopのコードを削っていき、再現してコードを作成しました。

回避方法は、発生状況に応じて各々作る必要があると思いますが、@Environment(\.dismiss)を避けるという方法があります。

他の方法は、上記リンクに記載があります。

### その他

iOS16.0以降NavigationStackはバグだらけだったので、iOS16.3まではβ版を使ったデバッグをしていたのですが、iOS16.4では落ち着いているだろうと油断したところ、iOS16.4のリリース後に遭遇してしまったバグです。

これは余談ですがNavigationStackは品質がとても低いままリリースされていて、iOS16.4時点では、使い方によっては実用に絶えないのではないかと考えています。
私は[Apple Developer Forum NavigationLink in NavigationStack doesn't work on iOS 16.0 and iOS 16.1 beta](https://developers.apple.com/forums/thread/715970)にPostした[回避策](https://github.com/hmuronaka/NavigationDestinationSelectedViewModifier)を利用していますが、iOS16.5時点では問題なく動作しています。

## バグ２　iOS16.0以降、Listのスクロールと、ListにBindingしているデータの更新が合わさるとクラッシュすることがある。(iOS16.4.1で解消)

[Apple Developer Forum　ScrollViewProxy scrollTo will crash when scrolling outside of bounds of previous (not current) List data source/array in iPad and iOS 16 beta](https://developers.apple.com/forums/thread/712510?answerId=729039022#729039022)

### 事象

iOS15では問題なくListのスクロールとデータ更新が同時に行われてもクラッシュしなかったのですが、iOS16以降で発生するようになりました。

Listに表示するデータを変更した直後に、下記コードのように、Listを表示する際にscrollToでリストの表示位置を変更するとクラッシュします。

```swift

var body: some View {
  ScrollViewReader { proxy in
    List {
      ForEach(data) {}
    }
    .onAppear() { proxy.scrollTo(...) }
    .onChange() { newValue in proxy.scrollTo(...) } // たまにクラッシュする。
  }
}

```

### 原因

Listの実装がUITableViewから置き換わった影響？

[iOS 16 Release Notes](https://developer.apple.com/documentation/ios-ipados-release-notes/ios-16-release-notes)には、"The implementation of list no longer uses UITableView. (81571203)"と記載されていて、これが影響しているのかもしれません。(確かUICollectionViewに置き換わったと記憶しています。)

### 対応

データ更新に伴うListの再描画時に、前回の状態が維持されていることに問題があると推測して、Listのインスタンスを一度破棄させることで前回状態を破棄させるという[回避方法を作成](https://developers.apple.com/forums/thread/712510?answerId=729039022#729039022)しました。
(ただしView構造によっては、上記の対応でもまだクラッシュが発生することがあるようです。)

またSwiftUIではiOSバージョンによって挙動が異なるため、「iOS15,iOS16.4以降では既存処理を行う」「iOS16.0~16.3では回避策を適用する」といったiOSバージョンによる処理分岐が必要となりました。

#### その他

この回避方法を作成する上で、 WWDC21の[Demystify SwiftUI](https://developer.apple.com/videos/play/wwdc2021/10022/)は、大変参考になりました。

SwiftUIを利用する際には、WWDC21の[Demystify SwiftUI](https://developer.apple.com/videos/play/wwdc2021/10022/)とWWDC23の[Demystify SwiftUI performance](https://developer.apple.com/videos/play/wwdc2023/10160)の視聴すると、開発のヒントになるかもしれません。

##　互換性問題１　NavigationViewのバグを放置して、iOS16でNavigationViewをdeprecatedにしてNavigationStack/NavigationSplitViewを導入

SwiftUIは、バグのあるAPIをdeprecatedとして、新しいiOSバージョンからの新APIでバグを対象することがあります。

公式にアナウンスされた訳ではありませんが、NavigationViewはそのように受け取ることができる例だと思います。

[Apple Developer Forum　SwiftUI NavigationView pops back when updating observableObject](https://developers.apple.com/forums/thread/693137) 

### 事象

iOS15のNavigationViewではView構造によっては、ランダムに遷移先ビューからpopされてしまう不具合があります。

これはリストをスクロールした時、@EnvironmentObjectの有無、@StateObjectの有無、データ更新のタイミング、iOSのフォントサイズ...などなど様々な条件が絡み発生するバグNavigationViewでこのバグに遭遇すると、原因が全く分からずとても厄介です。

リンク先の回避方法にある .navigationViewStyle(.stack) を適用すると回避できたように見えるのですが、上記の条件の組み合わせによっては、.stackを適用しても発生することを確認しています。

### 対応

iOS15では、popしてしまうのを受け入れるか、NavigationViewを使わない、または@EnvironmentObjectの利用を止めたり、View構造を調整することで、一時的に回避できることがあります。

根本的にはiOS16以降のNavigationStackを利用する必要があります。

cut & loopでは、iOS15ではNavigationView, iOS16ではNavigationStackを利用しています。
またiOS15,iOS16の互換性の点から、以下のようなラッパークラスを用いて対応しました。

```swift
public struct CLNavigationView<Destination: View>: View {
    
    @ViewBuilder
    public let destination: () -> (Destination)
    
    public init(@ViewBuilder destination: @escaping () -> (Destination)) {
        self.destination = destination
    }
    
    public var body: some View {
        if #available(iOS 16.0, macOS 13.0, *) {
            NavigationStack {
                self.destination()
            }
        } else {
            NavigationView {
                self.destination()
            }
            .navigationViewStyle(.stack)
        }
    }
}
```

## 互換性問題2　iOS16からiOS17のObservationへどのように以降すれば良いか分からない問題

これはバグではありませんが...

iOS16以前のObservedObject等に変わり、iOS17からObservationが導入されました。

WWDC23の[Demystify SwiftUI performance](https://developer.apple.com/videos/play/wwdc2023/10160)でも推奨されていることから、今後はObservationを利用した方が良いのでしょう。(詳細な理由は把握できていません。）

こちらの記事[Migrating from the Observable Object protocol to the Observable macro](https://developer.apple.com/documentation/swiftui/migrating-from-the-observable-object-protocol-to-the-observable-macro)にMigrationに関する内容がありますが、これはアプリのサポートバージョンをiOS17以降にした場合にのみ利用できる方法です。

下位互換を保ったまま、どのようにObservationを利用すれば良いのか、とても悩ましいです。

# 教訓

- iOSのリリースノートのSwiftUI関連は必ず確認すること。
- βのリリースノートに、利用APIに関する変更が記載されている場合、βが取れるまで継続的に動作検証すること。
- βの間のバグやWarningの修正は、状況に応じて取り込むこと。βの間に修正されるものも多い。（そのままのことも多いので、回避策は検討しておく方が良い）
- SwiftUIのAPIIは、サンプルプログラムなどを作成して、プロダクトの想定する使い方ができるか事前に確認すること。
